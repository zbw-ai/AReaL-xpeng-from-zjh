## 2. 确认有效的结论

### 2.1 AReaL RL pipeline 功能正确

- rollout → reward → train → weight sync → eval 全链路跑通
- 9 组训练无一因框架 bug 崩溃（崩溃全是训练不稳定导致）
- reward 函数修复后（deepmath 非数值答案），所有 answer 类型都能正确计算
- throughput/MFU 指标已实现（待集成测试验证）

### 2.2 Instruct+think 是确定的最佳模型设定

| 设定 | dapo_math task_rw | 训练稳定性 | 评测最佳 |
|---|---|---|---|
| Instruct nothink | 0.83（太高）→ 下降 | step 500 崩 | 76.77% |
| Instruct+think | 0.83（太高）→ 稳定更久 | step 600 崩 | 76.43% |
| SFT-4450 nothink | 0.77 → 下降 | step 600 崩 | 70.87% |

虽然评测分数差不多，但 Instruct+think 在训练中表现最稳定（entropy 爆炸最晚），且在 deepmath 上修复后初始 reward 最高（0.73）。

### 2.3 内部 SFT 非常有效

Base → SFT: 34.64% → 71.79%（+37%）。但 SFT-4450 在所有 RL 实验中都不如 Instruct——Instruct 的后训练（SFT + RLHF + GRPO）质量远高于内部 SFT。

### 2.4 Reward bug 影响巨大

deepmath v1 (bug): task_reward=0.19 vs v2 (修复后): task_reward=0.73。同模型同数据，差异完全来自 80% 非数值答案无法解析。**任何新数据集都必须先验证 reward 函数的覆盖率。**

## 3. 为什么 RL 没有提升模型

### 3.1 直接原因：数据难度不匹配

这是唯一的根本原因。

```
GRPO 的 reward 信号 = group 内正确与错误回复的 advantage 差
                    ↓
当正确率 >70%: 8 个回复中 5-6 个正确 → advantage 几乎全正 → 梯度信号弱
当正确率 <20%: 8 个回复中 7-8 个错误 → advantage 几乎全负 → 梯度信号也弱
                    ↓
理想区间 20-50%: 正确和错误大致均衡 → advantage 有清晰的对比信号
```

| 数据集 | Instruct 正确率 | 理想区间 | 问题 |
|---|---|---|---|
| dapo_math_17k | 83% | 20-50% | 太简单，83% 都对了 |
| deepmath_20k | 73% (修复后) | 20-50% | 偏高 |

### 3.2 间接原因：训练配置与业界最佳实践的差距

| 维度 | 我们的配置 | 业界最佳 (POLARIS) | 差距评估 |
|---|---|---|---|
| 数据 | 全量 17K/20K，未过滤 | 53K→30K 精选，按模型能力校准 | 致命差距 |
| 序列长度 | 8K response | 52K train, 90K infer | 限制长链推理 |
| 温度 | 0.99 | 1.4-1.5 | 多样性不足 |
| 算法 | GRPO (对称 clip, 有 KL) | DAPO+GRPO+ (Clip-Higher, 无 KL) | 探索受限 |
| 训练策略 | 单阶段 | 3 阶段递增温度+迭代过滤 | 过于简单 |

**这些间接原因不是导致失败的根因——即使配置完美，如果数据难度不匹配，RL 仍然不会有效。** 但解决数据问题后，这些差距会成为进一步提升的瓶颈。

### 3.3 不是原因的因素

| 排除项 | 证据 |
|---|---|
| 框架 bug | 9 组训练无框架错误 |
| Reward 计算 | 修复后 v2 正确率 73%，reward 计算正确 |
| 模型能力不足 | Instruct P0 基线 76.24%，能力足够 |
| GPU/显存不足 | 显存利用率 45-69%，未 OOM |

## 4. 如何才能训练出更好的模型

### 4.1 第一要务：让模型在训练集上"会一部分但做不稳"

**做法：数据难度校准**

```
Step 1: 用 Instruct+think 在候选数据集上每题 rollout 8 次
Step 2: 统计每题的通过率（0/8, 1/8, ..., 8/8）
Step 3: 过滤
  - 移除通过率 = 0%（太难，模型完全不会，没有正向信号）
  - 移除通过率 > 75%（太简单，模型已经会了，没有学习空间）
  - 保留通过率 10-60% 的题（模型"有时对有时错"= 最大 RL 信号）
Step 4: 用过滤后的数据集跑 RLVR
```

**预期效果：** POLARIS 用这个方法从 53K 过滤到 30K，Qwen3-4B 在 700 步内 AIME24 从 73.8% 提到 81.2%（+7.4%）。

**数据来源选择：**
- deepmath_20k 过滤后预期保留 3-8K（大部分题对 Instruct 太简单或答案格式不匹配）
- 可以合并 deepmath + dapo_math + 其他数据源，先合并再统一过滤
- 理想规模：10-30K 过滤后题目

### 4.2 第二步：拉长序列

```
当前: max_new_tokens = 8192, context_length = 16384
目标: max_new_tokens = 16384~32768, context_length = 32768
```

**为什么重要：**
- 难题需要更长的推理链。当前 8K 限制下 no_eos_ratio 达到 12.5%——模型思考被截断，答案不完整，reward=0
- POLARIS 训练 52K、推理 90K；我们只有 8K，差距 6 倍
- 拉长序列后，模型能在更多题上产出完整答案，有效 reward 信号增加

**风险：** 显存可能不够（A100 80G），需要测试。PP=2 已经为此预留了空间。

### 4.3 第三步：切换到 DAPO 算法

```yaml
# 在 YAML 中添加/修改：
actor:
  eps_clip: 0.2              # clip 下界（不变）
  eps_clip_higher: 0.28      # clip 上界（DAPO Clip-Higher）
  kl_ctl: 0.0               # 移除 KL（DAPO 标准）
  mask_no_eos_with_zero: true # 截断序列排除出 loss
dynamic_bs: true              # Dynamic Sampling
```

**为什么：**
- DAPO 的 Clip-Higher 替代 KL 来约束策略更新——不限制探索方向，只限制步幅
- Dynamic Sampling 过滤掉 reward 全相同的 group（正确率 >90% 的题在 group 内全对，无信号）
- Overlong Mask 把截断序列排除出 loss，避免噪声梯度
- 这些机制在 Run 2 的 entropy 爆炸中本可以起到保护作用

### 4.4 第四步：提高采样温度

```yaml
gconfig:
  temperature: 1.2  # 从 0.99 提高（POLARIS 用 1.4）
```

**为什么：**
- 温度越高，采样多样性越大，pass@k 上限越高
- RL 的增益空间 = pass@k - pass@1，温度低 → pass@k 接近 pass@1 → RL 无处可压缩
- POLARIS 用 1.4-1.5 的极高温度，配合数据过滤保证效率
- 但不要一步到 1.4——先 1.2 看稳定性，再逐步提升

### 4.5 执行顺序

```
Phase A: 数据校准（解决根本问题）
  1. 写离线 rollout 脚本，评估每题通过率
  2. 过滤构造精选数据集
  3. 用精选数据重跑 RLVR
  → 预期：reward 明显上升，eval 有提升

Phase B: 训练配置优化（提升天花板）
  4. 拉长序列 8K→16K+
  5. 切 DAPO 算法
  6. 提高温度 0.99→1.2
  → 每次只改一个变量，观察效果

Phase C: 多阶段训练（逼近 POLARIS 路线）
  7. 跑完第一轮 RL 后，重新评估通过率分布
  8. 过滤掉已掌握的题（通过率 >90%），补充新题
  9. 第二轮 RL（温度递增）
  → 迭代至收敛
```

## 5. 关键教训

### 5.1 数据 > 算法 > 超参数

所有失败的根因都是数据问题，不是算法或超参数。在数据难度不匹配的情况下，无论用 GRPO 还是 DAPO，无论温度 0.99 还是 1.4，RL 都不会有效。

**正确的优先级：先确保数据在理想难度区间，再调算法和超参。**

### 5.2 "模型太强"也是问题

Instruct 在 dapo_math 上正确率 83%——这不是优势，是劣势。RL 需要"做不稳"的题来学习。POLARIS 明确移除了通过率 >90% 的题。

**选更强的模型不一定更好；配套数据的难度校准更重要。**

### 5.3 Reward 函数必须先验证

deepmath 的 reward bug（80% 答案无法解析）浪费了一整轮实验。任何新数据集接入前，必须：
1. 检查 answer 字段分布（数值/文本/布尔）
2. 模拟 reward 计算，确认覆盖率 >95%
3. 端到端测试 10 条样本

### 5.4 评测噪声不可忽视

AIME 2024 只有 30 题，单题权重 3.3%。±1% 的变化完全可能是随机波动。需要多次评测取平均，或用更大的评测集（Math 500 更可靠）。

### 5.5 先跑通再优化

Run 1 的 59 步验证了 pipeline 可用，然后才开始正式实验——这个策略是对的。但应该在 Run 2 之前就做数据难度评估（评估模型在训练集上的正确率），而不是跑了 2000 步才发现数据太简单。
